A. COSAS QUE HEMOS PEDIDO / DEFINIDO / CONSTRUIDO Y NO ESTÁN CLARAS O NO APARECEN EN EL DOCUMENTO
============================================================

Estas son capacidades técnicas, de arquitectura o de plataforma que sí hemos trabajado en el hilo, pero que el documento funcional orientado a clínica no recoge de forma explícita o no desarrolla lo suficiente:

1. Arquitectura multi-tenant por organization + branch
- Migración conceptual de clinic_id hacia organization_id.
- Separación organization / branch / org_members.
- Scope de permisos por organización y sucursal.
- Posibilidad de que un mismo usuario pertenezca a varias organizaciones y varias sucursales con roles distintos.
- Regla de un solo owner por organization, sin copropiedad.

2. Modelo de membresías y permisos avanzados
- org_members con role y branch_scope.
- Contexto activo por organización al iniciar sesión.
- Soporte para asistentes compartidas entre varios odontólogos/organizaciones.
- Aislamiento estricto entre tenants por RLS.

3. Onboarding técnico y wizard de activación
- Ruta /onboarding y /onboarding/resume.
- onboarding_status con pasos del wizard.
- createOrganization con validación de duplicados.
- Redirección por middleware según sesión/estado.
- Step específico para conectar WhatsApp dentro del onboarding.
- Acción connectWhatsApp y opción markWhatsAppConnected como skip.

4. Seguridad y RLS a nivel plataforma
- Helpers RLS como is_org_member() y get_user_org_role().
- Validaciones para evitar leakage entre tenants.
- Guards de platform_role en rutas sensibles.
- Revisión explícita de rutas con service_role client + filtros organization_id.

5. Configuración técnica de WhatsApp multi-tenant
- branch_whatsapp_config por branch.
- phone_number_id como routing key entrante.
- access_token_enc cifrado por branch.
- waba_id por branch.
- webhook_routing_key para n8n.
- Variables globales en Vercel solo para WHATSAPP_VERIFY_TOKEN y WHATSAPP_APP_SECRET.
- Tokens operativos por clínica/sucursal en DB, no globales.

6. Webhook handler de WhatsApp
- app/api/whatsapp/webhook/route.ts.
- GET con hub.challenge para verificación de Meta.
- POST con validación HMAC SHA-256 usando WHATSAPP_APP_SECRET.
- Lectura del raw body antes del parse JSON.
- Resolución del branch por phone_number_id.
- Insert en webhook_queue.
- Respuesta 200 rápida.
- Observabilidad con logs para missing phone_number_id y no active branch.
- Extracción de eventId/wamid para trazabilidad.

7. Cifrado de credenciales WhatsApp
- Helper lib/crypto/whatsapp-token.ts.
- AES-256-GCM.
- IV aleatorio de 12 bytes.
- Formato iv:authTag:ciphertext.
- Variable WHATSAPP_TOKEN_ENC_KEY.
- Cifrado/descifrado solo en backend o Server Actions.

8. Estrategia de environment variables y despliegue
- Definición detallada de env vars de Supabase, Gemini, Anthropic, WhatsApp, n8n, cron y admin.
- Separación entre variables públicas, privadas y por branch.
- Riesgo de faltantes en Vercel para production readiness.
- Regeneración de types/database.ts desde Supabase CLI.

9. Gobernanza del modelo de IA
- Decisión arquitectónica de usar Gemini como modelo principal.
- Anthropic/Claude solo como legacy o fallback temporal.
- Separación entre IA semántica vs reglas puras.
- Regla: n8n orquesta, Supabase guarda estado, Gemini interpreta/clasifica.

10. Estado de calidad / auditoría técnica final
- Limpieza de referencias clinic_id activas.
- Eliminación de workflow_runs obsoleta.
- Fixes en analytics/signals, renewal, performance-alerts, intake-sla, reactivation.
- Revisión de deploy readiness.
- Identificación del webhook faltante como blocker de WhatsApp.

11. Plataforma admin / superadmin
- Mención de superadmin/plataforma como capa separada.
- Rutas internas admin protegidas.
- ADMIN_DASHBOARD_SECRET.
- Visión de operación multi-tenant de plataforma, no solo visión clínica.

12. QA y pruebas del onboarding / WhatsApp
- Plan de pruebas de happy path.
- Reanudación desde onboarding_status.
- Casos de error de WhatsApp token inválido, duplicados, RLS y flows incompletos.
- Recomendación de smoke tests E2E para onboarding + webhook_queue.

C. ZONA GRIS / PARCIALMENTE CUBIERTA
============================================================

Hay temas que sí están en el documento y también los hemos tocado en el hilo, pero más desde arquitectura que desde implementación funcional completa:

1. WhatsApp
- Sí se trabajó fuertemente la integración, seguridad, routing y onboarding.
- Pero el documento habla además de uso clínico/operativo dentro de bandeja omnicanal y automatizaciones ya visibles para el usuario.

2. Leads con IA
- Sí definimos que Gemini debe clasificar leads y ser el corazón del sistema.
- Pero no se ha pedido aún construir toda la UI/UX del clasificador, score visible, colas de priorización o panel de revisión.

3. Dashboard
- Sí hablamos de métricas, renewal, alerts, analytics y plataforma.
- Pero el documento clínico plantea un dashboard gerencial mucho más rico y orientado a usuario final.

4. Roles y permisos
- Sí se diseñó arquitectura RBAC multi-org/multi-branch.
- Pero no se ha pedido aún la matriz funcional completa de permisos por pantalla/acción para admin, gerente, recepcionista, médico y marketing.
